## Work Done

### Fixed 4 Macro Test Failures

After adding V30MZ, M68000, and ARM7TDMI mnemonics to `Lexer.IsMnemonic()` in the previous session, 4 macro-related tests were still failing:

- `MacroExpansion_WrongArgumentCount_ReportsError`
- `MacroDefaultParameter_Output_PartialOverride`
- `MacroDefaultParameter_Output_MultipleDefaults`
- `MacroSystem_Output_DataDirectiveInMacro`

**Root cause**: `b` (ARM branch instruction) classified as Mnemonic, but `Parser.ParseMacroDefinition()` only accepted `TokenType.Identifier` for parameter names. Macros with parameter `b` (e.g., `.macro load_two a=$10, b=$20`) had their `b` parameter silently dropped.

**Fix**: Extended the parameter parsing loop in `ParseMacroDefinition()` (line 424) to accept both `Identifier` and `Mnemonic` tokens as parameter names:

```csharp
while (Check(TokenType.Identifier) || Check(TokenType.Mnemonic)) {
```

### Known Impact

The single-letter `b` (ARM branch) is the most problematic mnemonic because it conflicts with common parameter/label names. The parser fixes handle this:

- `b:` (label definition) → parsed as label via mnemonic-as-label lookahead
- `.macro foo a, b` → `b` accepted as parameter via mnemonic parameter fix
- `b target` (ARM instruction) → parsed as instruction (no colon after)
